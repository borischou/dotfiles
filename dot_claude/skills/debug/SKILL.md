# Systematic Debugging

遇到 bug 时的强制流程。**Rigid skill** — 严格按步骤执行，不跳步。

## 核心原则

- **找根因，不修表象** — 补丁式修复是技术债的来源
- **证据驱动** — 每个结论必须有日志/堆栈/复现步骤支撑
- **最小改动** — 修复只改必要的代码，不顺手重构

## 流程

### Step 1: 复现与定义

明确两件事：

1. **实际行为**：具体的错误信息、堆栈追踪、异常输出
2. **预期行为**：正确情况下应该发生什么

如果无法复现，先搞清楚复现条件再继续。不要在无法复现的情况下猜测修复。

输出格式：
```
症状: [具体描述]
预期: [应该怎样]
复现: [步骤或条件]
```

### Step 2: 定位

使用以下技术缩小范围（按效率排序）：

1. **读错误信息** — 堆栈追踪直接指向问题？先读完再动手
2. **搜索关键字** — 用 Grep 搜索错误信息、异常类名、关键变量
3. **追踪数据流** — 从出错点向上追溯，数据在哪一步变坏了？
4. **二分法** — 对于复杂流程，在中间点加日志，确认上半段还是下半段出问题
5. **git bisect** — 如果是回归 bug，用 bisect 找到引入问题的 commit

**禁止**：不读代码就猜原因、不看堆栈就改代码、同时改多处碰运气。

### Step 3: 假设与验证

1. 列出 **最多 3 个** 可能原因，按可能性排序
2. 对每个假设设计一个最小验证方法（加日志、改输入、跑特定测试）
3. 逐个验证，排除到只剩一个

格式：
```
假设 1: [描述] → 验证方式: [操作] → 结果: [确认/排除]
假设 2: [描述] → 验证方式: [操作] → 结果: [确认/排除]
```

### Step 4: 修复

确认根因后执行修复：

1. **最小改动** — 只改导致 bug 的代码
2. **解释 why** — 在 commit 或注释中说明为什么这样改，而不是改了什么
3. **不顺手重构** — 修 bug 就是修 bug，周边代码的"改进"另开任务

### Step 5: 验证

1. 运行之前失败的测试/操作，确认修复
2. 运行全量测试，确认没有引入新问题
3. 如果修复涉及边界条件，手动验证边界 case

**测试必须通过才能继续。** 如果测试失败，回到 Step 2。

### Step 6: 防御

评估是否需要加防护：

- **需要加测试的情况**：bug 涉及的逻辑路径之前没有测试覆盖
- **需要加断言的情况**：bug 源于非法状态/数据，应在入口处拦截
- **不需要加的情况**：已有测试覆盖、纯 typo、一次性配置错误

如果需要，写测试/断言。不需要则跳过，不要为了"完整"而加无用的测试。

### Step 7: 记录

调用 `/lessons` 记录本次调试经验（如果 `/lessons` skill 可用）：

- 问题描述
- 根因
- 修复方式
- 防范规则（下次怎么避免）

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 看到报错就改代码 | 先读完堆栈，理解调用链 |
| 同时改多处 "试试" | 一次只改一处，验证后再改下一处 |
| 修复后不跑测试 | 每次修复后立即跑测试 |
| 修 bug 顺手重构 | bug 修复和重构分开提交 |
| 加 try-catch 吞异常 | 找到异常根因，而非掩盖 |
| "在我这跑得通" | 确认环境差异，在目标环境验证 |

## 浏览器自动化专项

针对 maimai/boss 项目的常见 bug 类型：

| 症状 | 常见根因 |
|------|---------|
| 搜索结果为空 | SPA 未加载完成，需等待特定元素而非 page load |
| profile_url 为空 | contactcard 链接结构变化，检查 JS 提取逻辑 |
| 登录丢失 | Cookie 过期/被踢，需检查登录状态后再操作 |
| 消息发送失败 | 配额耗尽但错误被吞，检查 quota 相关异常处理 |
| 页面导航超时 | 重试逻辑未覆盖该路径，检查 retry_navigation |
