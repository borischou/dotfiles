# Code Review Memory

带记忆的工程审查：分析前回顾历史，修复后自动复盘归档。适用于任意语言和框架的项目。

## 历史记录文件

路径：项目根目录下 `.claude/review-history.md`

如果文件不存在，在 Phase 1 创建初始文件：

```markdown
# Code Review History

工程审查历史记录，由 code-review-memory skill 自动维护。

---
```

## 测试命令自动检测

不硬编码测试命令，按以下优先级自动检测项目的测试方式：

1. 查 `package.json` 中的 `scripts.test` → `npm test` 或 `yarn test` 或 `pnpm test`（看 lockfile 判断包管理器）
2. 查 `pyproject.toml` 或 `pytest.ini` 或 `setup.cfg` 中的 pytest 配置 → `pytest tests/ -v`
3. 查 `Makefile` 中是否有 `test` target → `make test`
4. 查 `Cargo.toml` → `cargo test`
5. 查 `go.mod` → `go test ./...`
6. 查 `build.gradle` 或 `pom.xml` → `./gradlew test` 或 `mvn test`
7. 以上都没有 → 询问用户项目的测试命令

检测结果在 Phase 2 首次使用前告知用户确认。

## 工作流程

### Phase 1: 回顾历史

1. 读取项目根目录下 `.claude/review-history.md`
2. 如果有历史记录，向用户展示摘要：
   - 历史审查次数
   - 上次审查日期和主要发现
   - 反复出现的问题模式（如果有）
   - 上次遗留的待办项（如果有）
3. 如果没有历史记录，告知用户这是首次审查

### Phase 2: 分析与修复

1. 使用 Explore agent 全面扫描工程，关注：
   - 代码缺陷（bug、逻辑错误、异常处理不当）
   - 安全隐患（硬编码凭证、注入风险）
   - 性能问题（N+1 查询、不必要的重复计算）
   - 架构问题（循环依赖、职责不清）
   - 代码风格（不一致的命名、重复代码）
   - 配置问题（硬编码值、缺少校验）
2. **对照历史记录**，标记哪些是新问题、哪些是历史遗留
3. 向用户报告发现，按严重程度排序（严重 > 中等 > 轻微）
4. 用户确认后执行修复
5. 修复完成后运行测试验证（使用自动检测的测试命令）

### Phase 3: 死代码清理

修复完成后，专门做一轮死代码/无用代码扫描，避免补丁累积产生冗余：

1. 扫描以下类型的死代码：
   - 未被调用的函数/方法/类
   - 未使用的 import/require/include
   - 被注释掉的代码块（超过 3 行的注释代码）
   - 已废弃但未删除的变量/常量
   - 空的异常处理块（无实际逻辑）
   - 仅为向后兼容保留但已无调用方的代码（如旧接口别名、重导出）
2. 对照本次 Phase 2 的修复内容，重点检查修复是否引入了新的冗余（如旧实现未删、临时调试代码残留）
3. 向用户报告发现的死代码，说明每项为什么判定为无用
4. 用户确认后执行清理
5. 清理完成后再次运行测试

### Phase 4: 生成复盘记录

修复和清理全部完成后，将以下内容**追加**到 `.claude/review-history.md` 末尾：

```markdown
## Review #{序号} - {YYYY-MM-DD}

### 发现的问题
| 严重程度 | 文件 | 问题描述 | 状态 |
|---------|------|---------|------|
| 严重/中等/轻微 | path/to/file:行号 | 问题简述 | 已修复/待处理 |

### 修复内容
- 简述每个修复的改动和原因

### 死代码清理
| 文件 | 清理内容 | 类型 |
|------|---------|------|
| path/to/file:行号 | 清理简述 | 未调用函数/未使用import/注释代码/废弃变量/空块/旧兼容 |

### 测试结果
- 测试运行结果摘要

### 复盘
- **做得好的**：本次修复中值得保持的做法
- **待改进的**：仍需关注的问题或下次应注意的点
- **遗留项**：未处理的问题及原因

---
```

## 注意事项

- 不要重复修复历史记录中已标记"已修复"的问题，除非它复发了
- 复盘记录用中文编写
- 每条记录都要包含具体文件路径和行号，便于追溯
- 如果用户只想分析不想修复，Phase 2 中跳过修复步骤，Phase 4 中所有问题标记为"待处理"
